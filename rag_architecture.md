# 气象 App 客服 RAG 架构设计

## 1. 场景定位

**场景**：气象 App 用户反馈诊断助手
**数据规模**：171 条结构化 feedback（持续增长）
**复杂度**：跨模块、多维度、需要气象专业知识

### 为什么不是简单规则 JSON

| 规则 JSON 能做 | 本场景需要但做不了 |
|---------------|-------------------|
| 精确关键词匹配 | 「热死了」≈「温度比预报高5度」，规则匹配不到 |
| 单一分类查询 | 「预警没收到」同时涉及 `天气预警` + `App体验`(推送) |
| 返回固定答案 | 需组合多条 feedback + 实时数据源才能回答 |
| 静态维护 | 171 条且增长，规则冲突和覆盖盲区指数恶化 |

**结论**：规则 JSON 仅做最外层兜底（如「怎么开推送」→ 直接返回 SOP），复杂反馈必须上 RAG。

## 2. 知识库设计：三层索引

### 2.1 数据模型 (7 字段)

```
module          → Layer 1: 产品模块，决定进入哪个子索引
problem_pattern → Layer 2: 问题模式，子索引内粗筛
problem_desc    → 用户反馈原文表述
root_cause      → 技术根因分析
solution        → 改进/解决方案
tags            → Layer 3: 多维语义标签
severity        → 严重程度，排序加权
```

### 2.2 Layer 1: 产品模块 (14 类)

```
┌──────────────────────────────────────────────────────────┐
│  Module          │  覆盖范围                              │
├──────────────────┼────────────────────────────────────────┤
│  定位            │  GPS/基站定位、区域识别、异地切换       │
│  实况            │  当前温/风/湿/压/能见度/天气现象/       │
│                  │  UV/体感/闪电/浪高/闷热               │
│  逐小时预报      │  未来48h逐时温度/风速/降水/湿度        │
│  逐天预报        │  3-15天日预报、日出日落               │
│  短临降水        │  0-2h 雷达外推降雨、分钟级降水         │
│                  │  到达/结束时间                        │
│  格点可视化      │  雷达图/热力图/预报图层/拼图/色标     │
│  天气预警        │  台风/暴雨/高温/寒潮/大风/雷电/        │
│                  │  冰雹/暴雪/洪水/地质灾害等几十种       │
│                  │  生成→发布→接收→解除全链路            │
│  空气质量        │  AQI/PM2.5/PM10/O₃/雾/霾/沙尘/       │
│                  │  秸秆焚烧/室内污染                    │
│  台风预报        │  路径/强度/登陆点/影响范围/集合预报    │
│  潮汐海浪        │  潮汐/浪高/海温/涌浪/海雾/赤潮        │
│  生活指数        │  运动/晨练/穿衣/洗车/钓鱼/晾晒/       │
│                  │  空调/花粉/中暑                      │
│  航空专业气象    │  航路天气/颠簸/风切变/积冰/大雾/      │
│                  │  无人机/通航                          │
│  数据时效        │  各模块更新延迟、页面不一致、          │
│                  │  海外/特殊区域刷新                    │
│  App体验         │  闪退/耗电/Widget/通知/多城市/        │
│                  │  语音播报/Push策略                    │
└──────────────────────────────────────────────────────────┘
```

### 2.3 Layer 2: 问题模式 (5 类)

```
┌─────────────────┬──────────────────────────────────────┬──────────┐
│  problem_pattern │  判定标准                             │  占比     │
├─────────────────┼──────────────────────────────────────┼──────────┤
│  数据偏差        │  观测/预报值与实际不符                 │  ~40%    │
│                 │  (温度差N度、风力偏小、湿度不对等)       │          │
│  时空误差        │  时间不准(早了/晚了)                   │  ~20%    │
│                 │  空间不准(落区偏、站点稀疏)             │          │
│  时效延迟        │  数据/预警/推送更新不及时              │  ~15%    │
│  认知偏差        │  用户不理解气象原理导致误判            │  ~15%    │
│  体验缺陷        │  App本身的bug/性能/交互/Push策略问题   │  ~10%    │
└─────────────────┴──────────────────────────────────────┴──────────┘
```

### 2.4 Layer 3: Tags

自由维度标签，格式: `主标签, 子标签, ...`

标注维度:
- **天气要素**: 温度/风速/降水/湿度/气压/能见度/紫外线/雷电/闪电
- **天气系统**: 台风/暴雨/强对流/寒潮/高温/大雾/霾/沙尘/冰雹/暴雪
- **技术栈**: 雷达外推/数值模式/集合预报/卫星反演/光流法
- **用户场景**: 出行/运动/上班/旅游/户外作业
- **影响等级**: 生命安全/财产损失/生活不便/体验不佳

## 3. RAG Pipeline 架构

### 3.1 整体链路

```
用户输入: "我这儿明明大暴雨，App还说小雨，怎么回事？"
              │
     ┌────────▼────────┐
     │ 1. Query Router  │  ← 意图路由
     │   模型: 小模型    │     分类: 反馈诊断 / 功能操作 / 闲聊
     │   延迟: <50ms     │
     └────────┬────────┘
              │ feedback类
     ┌────────▼────────┐
     │ 2. Module Router │  ← Layer 1 路由
     │   模型: 中模型    │     输出: {module: "实况", confidence: 0.93}
     │   延迟: <100ms    │     可多标签, 支持 cross-module 召回
     └────────┬────────┘
              │ module: 实况
     ┌────────▼────────┐
     │ 3. Hybrid Search │  ← 混合召回 (三路并行)
     │   - 结构化过滤    │     WHERE module='实况' AND pattern='数据偏差'
     │   - Dense Vector │     problem_desc+root_cause+solution embedding
     │   - Sparse BM25  │     关键词倒排索引 (暴雨/小雨/流量)
     │   延迟: <200ms    │
     └────────┬────────┘
              │ Top-K 候选 (10~20条)
     ┌────────▼────────┐
     │ 4. Reranker      │  ← 精排
     │   Cross-encoder   │     BGE-Reranker / Cohere Rerank
     │   延迟: <100ms    │     过滤无关, 保留 Top-3~5
     └────────┬────────┘
              │ 最相关的 3 条
     ┌────────▼────────┐
     │ 5. LLM Generator │  ← 生成回答
     │   模型: 大模型    │     注入:
     │   延迟: <2s       │     - Top-3 feedback(root_cause + solution)
     │                  │     - 当前实时天气数据 (可选 MCP 注入)
     │                  │     - 客服话术风格约束
     └────────┬────────┘
              │
         最终回答
         ├── 共情话术
         ├── 根因解释 (基于 root_cause)
         ├── 现状说明 (基于实时数据)
         └── 改进承诺 (基于 solution)
```

### 3.2 各环节技术选型

```
┌─────────────────┬──────────────────────┬──────────────────────────┐
│     环节         │      技术方案          │         说明              │
├─────────────────┼──────────────────────┼──────────────────────────┤
│ Query Router    │ LLM few-shot +       │ 3~5类意图分类              │
│ (意图分类)       │ constrained output   │ 延迟: <50ms               │
├─────────────────┼──────────────────────┼──────────────────────────┤
│ Module Router   │ LLM + structured     │ 输入14个module定义         │
│ (模块路由)       │ prompt + JSON output │ 支持多标签输出             │
│                 │ 或 embedding 相似度    │ 延迟: <100ms              │
├─────────────────┼──────────────────────┼──────────────────────────┤
│ Hybrid Search   │ pgvector             │ 三路并行召回               │
│ (混合召回)       │ + BM25 (或Elastic)   │ structured filter          │
│                 │                      │ + dense vector             │
│                 │                      │ + sparse keyword           │
│                 │                      │ 延迟: <200ms              │
├─────────────────┼──────────────────────┼──────────────────────────┤
│ Reranker        │ BGE-Reranker-v2      │ Cross-encoder精排          │
│ (重排序)         │ 或 Cohere Rerank      │ 延迟: <100ms              │
├─────────────────┼──────────────────────┼──────────────────────────┤
│ Generator       │ DeepSeek / GPT-4o /  │ Prompt工程核心             │
│ (生成)           │ Claude               │ 延迟: <2s                 │
├─────────────────┼──────────────────────┼──────────────────────────┤
│ Embedding       │ BGE-large-zh /       │ 中文语义向量化             │
│                 │ text2vec-large-chinese│                          │
├─────────────────┼──────────────────────┼──────────────────────────┤
│ 评估 & 闭环      │ RAGAS 框架           │ 检索命中率/答案有用率      │
│                 │ + LLM-as-Judge       │ Badcase自动回写            │
│                 │ + 用户👍👎反馈        │ 知识库持续迭代             │
└─────────────────┴──────────────────────┴──────────────────────────┘
```

## 4. 数据流转

### 4.1 知识入库 (Offline)

```
CSV (171条, 持续增长)
    │
    ├──→ 结构元数据 → PostgreSQL
    │    (module, problem_pattern, severity, tags[文本数组], created_at, updated_at)
    │
    ├──→ 文本拼接 → Embedding → pgvector
    │    concat(problem_desc, "\n", root_cause, "\n", solution, "\n", tags)
    │    → BGE-large-zh → 1024维向量
    │
    └──→ 倒排索引 → BM25 / Elasticsearch
         problem_desc + tags 分词建倒排
```

### 4.2 在线检索 (Online)

```
Query: "大暴雨App还报小雨"
  │
  ├─ Struct Filter: WHERE module='实况' AND problem_pattern IN ('数据偏差','时效延迟')
  │    → 候选池从171 → ~50条
  │
  ├─ Dense Vector: cosine_similarity(query_embedding, doc_embeddings)
  │    → Top-20
  │
  ├─ Sparse BM25: BM25(query, doc_texts)
  │    → Top-20
  │
  └─ Fusion: RRF (Reciprocal Rank Fusion) 合并 → Top-15
       → Reranker → Top-3
```

## 5. Prompt 工程

### 5.1 Module Router Prompt

```
你是气象App的智能路由助手。根据用户反馈，判断属于哪个产品模块。

可选模块及定义:
- 定位: GPS/基站定位偏差、区域识别错误
- 实况: 当前温度/风速/湿度/气压/天气现象/体感温度等观测数据
- 逐小时预报: 未来48h逐时预报
- 逐天预报: 3-15天日预报、日出日落
- 短临降水: 0-2h雷达降雨预测
- 格点可视化: 雷达图/热力图/预报图层
- 天气预警: 各类预警的发布/收到/解除/分级
- 空气质量: AQI/PM2.5/PM10/雾/霾/沙尘
- 台风预报: 台风路径/强度/影响范围
- 潮汐海浪: 潮汐/浪高/海温
- 生活指数: 运动/穿衣/洗车/钓鱼/花粉等指数
- 航空专业气象: 航路天气/颠簸/风切变
- 数据时效: 数据更新时间/各页面不一致
- App体验: 闪退/耗电/Widget/通知/语音播报

输入: "{用户query}"
输出JSON: {"module": ["主要模块", "次要模块(可选)"], "confidence": 0.0~1.0}
```

### 5.2 Generator Prompt

```
你是气象App的客服助手，回答用户关于天气数据的疑问。

## 性格
- 专业但不说教
- 共情先于解释
- 结论先行，详因在后

## 参考知识 (内部，不对用户展示)
{retrieved_feedbacks}

## 当前天气 (如有)
{live_weather_data}

## 规则
1. 先共情，理解用户的困惑
2. 用通俗语言解释根因（不要堆砌术语）
3. 告诉用户我们正在/将要怎么改进
4. 如果涉及用户设置问题（如推送未开），引导操作

## 输出
- 2-4句话
- 自然，像真人客服
- 结尾可加一个相关建议

用户问题: {query}
```

## 6. 评估体系

### 6.1 离线评估 (RAGAS)

| 指标 | 含义 | 目标 |
|------|------|------|
| Context Precision | 检索到的文档中相关比例 | >0.85 |
| Context Recall | 所有相关文档被检索到的比例 | >0.90 |
| Faithfulness | 生成答案是否忠于上下文 | >0.95 |
| Answer Relevance | 答案与问题相关度 | >0.85 |

### 6.2 在线评估

```
用户查询 → 生成回答 → 用户操作
                         │
                    ┌────┴────┐
                    ▼         ▼
                  👍 有用    👎 没用
                               │
                         ┌─────▼──────┐
                         │ 标记 badcase │
                         │ 人工复核     │
                         │ 回写知识库   │
                         └────────────┘
```

### 6.3 Badcase 闭环

```
badcase 触发
    │
    ├── 检索层 badcase → 补充/修改 tags + problem_desc
    │
    ├── 生成层 badcase → 优化 Generator Prompt
    │
    └── 知识库盲区 → 新增 feedback 条目入库
```

## 9. CSV 与 RAG 的对应关系

| CSV 字段 | 入库位置 | 检索阶段 |
|----------|---------|---------|
| `module` | PostgreSQL 结构化列 | Layer 1: WHERE 过滤 |
| `problem_pattern` | PostgreSQL 结构化列 | Layer 2: WHERE 粗筛 |
| `problem_desc` | Embedding 文本 | Dense/Sparse 语义匹配 |
| `root_cause` | Embedding 文本 | Dense 语义匹配 + Generator 注入 |
| `solution` | Embedding 文本 | Dense 语义匹配 + Generator 注入 |
| `tags` | PostgreSQL 数组 + Embedding 文本 | 结构化过滤 + 语义增强 |
| `severity` | PostgreSQL 结构化列 | Rerank 阶段: severity=high 加权 1.2x |
