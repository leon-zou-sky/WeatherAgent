## 一、项目概述

> 基于 RAG + Agent 架构的气象负反馈自动分析系统，用户反馈进来后，自动查询气象数据、检索历史案例、调用 LLM 生成分析报告和回复内容。

### 1.2 业务价值

|------|--------|--------|
| 排查时间 | 2小时/个 | 5分钟/个 |

| 响应速度 | 2-3天 | 实时 |

### 1.3 技术栈

Python 3.11 + FastAPI + 豆包大模型(Ark) + Milvus + MySQL + BGE-large-zh-v1.5
```

## 二、架构设计

### 2.1 整体架构

```
用户反馈 → API 层 → Agent 层 → Skill 层 → 数据层
                         ↓
                    LLM 分析 → 返回结果
```

**分层说明**：
- **API 层**：FastAPI 路由，接收反馈请求
- **Agent 层**：编排 Skill 调用，组装数据给 LLM
- **Skill 层**：独立功能模块（数据查询、知识检索、体感计算）
- **数据层**：MySQL（气象数据）、Milvus（知识库向量）

### 2.2 为什么选 Agent + Skill 架构

**面试话术**：

> "我们用的是 Skill + Agent 混合架构。Agent 负责智能决策——理解用户反馈、决定调用哪些 Skill、组合结果生成回复。Skill 负责固定功能——数据查询、链路检查、计算工具。
>
> 这样做的好处是：Skill 可以独立开发测试，Agent 可以灵活编排，后续新增数据源只需要加 Skill，不用改 Agent 逻辑。"

### 2.3 为什么没用 Function Calling

**面试话术**：

> "我们评估后选择了直接编排的方式，而不是 LLM Function Calling。原因是：
> 1. 我们的 Skill 调用顺序是固定的，不需要 LLM 动态决策
> 2. 直接编排延迟更低，一次 LLM 调用就够了
> 3. Function Calling 需要多次 LLM 调用（决策 + 生成），成本翻倍
>
> 如果后续 Skill 种类增多、需要动态选择，再引入 Function Calling。"

## 三、RAG 实现

### 3.1 什么是 RAG

**面试话术**：

> "RAG 是 Retrieval Augmented Generation 的缩写，检索增强生成。核心思路是：先从知识库中检索相关文档，拼入 LLM 的 Prompt，让 LLM 基于这些上下文生成回答。
>
> 解决的问题是：LLM 的知识有截止日期，而且不了解我们业务的历史案例。RAG 让 LLM 能参考最新、最相关的知识。"

### 3.2 我们的 RAG 流程

```
用户反馈: "温度比预报高5度"
    ↓
1. Embedding: 文本转向量 [0.12, -0.34, ...]
    ↓
2. Milvus 检索: 找最相似的 Top-3 历史案例
    ↓
3. 拼入 Prompt: "参考知识: {检索结果}"
    ↓
4. LLM 生成: 结合实时数据 + 历史案例，生成回复
```

### 3.3 为什么选 BGE-large-zh-v1.5

**面试话术**：

> "选型考虑了几个维度：
> 1. **中文优化**：BGE 是智源研究院专门为中文训练的，在中文 MTEB 排行榜上排名靠前
> 2. **开源免费**：MIT 协议，不需要付费调 API
> 3. **本地运行**：不依赖外部服务，延迟可控
> 4. **维度合适**：1024 维，精度比 768 维的小模型好
> 5. **生态成熟**：sentence-transformers 直接支持，社区活跃"

### 3.4 向量检索 vs 关键词匹配

| 维度 | 关键词匹配 | 向量检索 |
|------|-----------|---------|
| 原理 | 字符串包含 | 语义相似度 |
| "热死了" → "温度偏高" | ❌ 匹配不上 | ✅ 语义相近 |
| "又淋雨了" → "降雨偏差" | ❌ 匹配不上 | ✅ 语义相近 |
| "温度比预报高" → 精确匹配 | ✅ 能匹配 | ✅ 能匹配 |
| 实现复杂度 | 低 | 中 |
| 依赖 | 无 | Embedding 模型 + 向量数据库 |

**面试话术**：

> "我们实现了两层检索：向量检索优先，关键词匹配兜底。向量检索解决语义匹配问题——用户说'热死了'能匹配到'温度偏高'的案例。关键词匹配作为降级方案，确保向量服务不可用时系统仍然可用。"

### 3.5 知识库数据结构

```csv
module,problem_pattern,problem_desc,root_cause,solution,tags,severity
实况,数据偏差,温度比预报高5-10度,阳光直射体感差异,增加体感温度显示,温度,medium
```

**字段说明**：
- `module`：产品模块（14类）— 结构化过滤
- `problem_pattern`：问题模式（5类）— 分类匹配
- `problem_desc`：问题描述 — 向量检索主字段
- `root_cause`：根因分析 — 向量检索增强 + LLM 参考
- `solution`：解决方案 — 返回给 LLM
- `tags`：语义标签 — 向量检索增强
- `severity`：严重程度 — 排序加权

**Embedding 拼接**：`problem_desc + root_cause + solution + tags`

## 四、评估指标

### 4.1 核心指标

| 指标 | 定义 | 公式 | 我们的结果 |
|------|------|------|-----------|
| **准确率 (Precision)** | 检索结果中相关的比例 | 相关∩检索 / 检索 | - |
| **召回率 (Recall)** | 所有相关文档中被检索到的比例 | 相关∩检索 / 相关 | - |
| **Top-K 命中率** | Top-K 结果中包含正确答案的比例 | 命中数 / 总数 | 93.3% |
| **MRR** | 正确答案排名的倒数平均 | 1/rank 的平均 | - |

### 4.2 面试常见问题

**问：召回率和准确率怎么平衡？**

> "看业务场景。客服场景**召回率更重要**——宁可多返回几条相关案例给 LLM 参考，也不能漏掉关键信息。所以我们用 Top-3 而不是 Top-1。
>
> 但如果场景是精确匹配（比如法律条款检索），**准确率更重要**，返回不相关的结果会有误导。"

**问：怎么评估 RAG 效果？**

> "我们分三层评估：
> 1. **离线评估**：准备标注好的测试用例，跑检索看 Top-K 命中率和 MRR
> 2. **在线评估**：用户反馈 👍/👎 收集真实效果
> 3. **Badcase 闭环**：分析错误案例，补充知识库或优化检索策略"

**问：检索效果不好怎么优化？**

> "几个方向：
> 1. **优化 Embedding 模型** — 换更大的模型或领域微调
> 2. **优化知识库数据** — 补充更多案例，细化标签
> 3. **混合检索** — 向量 + BM25 + 结构化过滤，RRF 融合
> 4. **加 Reranker** — 用 Cross-encoder 精排
> 5. **优化查询** — Query 改写、扩展"

### 4.3 我们的评估脚本

```python
# 测试用例
TEST_CASES = [
    {"query": "温度比预报高5度", "expected_module": "实况"},
    {"query": "没收到预警", "expected_module": "天气预警"},
    {"query": "热死了", "expected_module": "实况"},
    ...
]

# 评估逻辑
for case in TEST_CASES:
    results = search_knowledge(case["query"], top_k=3)
    modules = [extract_module(r) for r in results]
    hit = case["expected_module"] in modules
```

## 五、技术细节

### 5.1 Embedding 是什么

**面试话术**：

> "Embedding 是把文本变成一组数字（向量）。语义相近的文本，向量距离也近。
>
> 比如'温度比预报高'和'太热了'，虽然字面完全不同，但 Embedding 后向量很接近，就能匹配上。
>
> 我们用的是 BGE-large-zh-v1.5，输出 1024 维向量。"

### 5.2 Milvus 是什么

**面试话术**：

> "Milvus 是专门做向量检索的数据库。传统数据库做精确匹配，Milvus 做相似度匹配。
>
> 我们用的是 Milvus Lite 版本，数据存在本地文件，适合开发和小规模场景。生产环境可以用 Milvus Standalone 或分布式部署。"

### 5.3 余弦相似度

**面试话术**：

> "余弦相似度衡量两个向量的方向差异。值在 -1 到 1 之间，1 表示完全相同，0 表示无关，-1 表示相反。
>
> 我们用余弦相似度衡量用户查询和知识库条目的相似程度，取 Top-3 最相似的。"

### 5.4 Prompt 工程

**面试话术**：

> "我们的 Prompt 分三部分：
> 1. **System Prompt**：定义 Agent 角色、能力、输出格式
> 2. **数据摘要**：实况数据 + 体感分析 + 逐时预报 + 知识库检索结果
> 3. **用户反馈**：原始的反馈内容
>
> LLM 综合这些信息，生成分析报告和回复。关键是把实时数据和历史案例都给到 LLM，让它有足够上下文。"

## 十、高含金量面试题（详细版）

### 10.1 你们怎么测试 RAG 检索效果？

**回答**：

> "我们分三层测试：
>
> **第一层：离线评估脚本**
> 准备了 15 个标注好的测试用例，每个用例包含用户查询和期望匹配的模块。跑完后计算 Top-1 和 Top-3 命中率。目前结果是 Top-1 命中率 93.3%。
>
> 测试用例覆盖了主要场景：温度偏差、降雨偏差、预警推送、体感温度、空气质量等。也有边界 case，比如用户说法模糊（'热死了'）或跨领域（'又淋雨了'）。
>
> **第二层：Badcase 分析**
> 脚本会自动输出 Badcase——那些检索结果和期望不符的用例。我们逐条分析原因：是知识库缺少相关案例？还是 Embedding 语义理解不够？还是标签分类有问题？
>
> 比如我们发现'又淋雨了'匹配到了'天气预警'而不是'实况'，分析后发现是因为知识库里'降雨偏差'案例的 module 标注不一致，修正后命中率提升了。
>
> **第三层：在线评估**
> 上线后通过用户反馈（👍/👎）收集真实效果。如果用户点了👎，标记为 Badcase，人工复核后回写知识库。"

**追问：15 个用例够吗？**

> "对于 MVP 阶段够了，覆盖了主要场景。后续会扩充到 50-100 个，覆盖更多边界 case 和长尾场景。工业界一般会维护一个'黄金测试集'，每次知识库更新后都跑一遍回归测试。"

---

### 10.2 LLM 生成的内容不准怎么办？怎么防止幻觉？

**回答**：

> "我们从三个层面控制 LLM 输出质量：
>
> **1. Prompt 约束**
> System Prompt 明确要求 LLM 只基于提供的数据回答，不要编造。输出格式是结构化 JSON，强制 LLM 按字段填充。
>
> **2. 数据注入**
> 把实时气象数据和检索到的历史案例都注入 Prompt，给 LLM 足够的上下文。这样 LLM 是'参考数据生成'而不是'凭空想象'。
>
> **3. 后处理校验**
> 解析 LLM 输出时，检查关键字段是否合理。比如温度值应该在合理范围内（-50℃~60℃），如果超出就标记异常。
>
> **4. 人工兜底**
> 严重问题（如预警、灾害相关）的回复需要人工审核后才能发送给用户。"

**追问：如果 LLM 忽略了检索到的知识怎么办？**

> "这种情况确实可能发生。我们的做法是：
> 1. 在 Prompt 中明确指示'请参考以下知识回答'
> 2. 检索结果放在 Prompt 的显眼位置
> 3. 如果多次出现 LLM 忽略知识的情况，说明知识和问题的相关性不够，需要优化检索"

### 10.3 你们的系统怎么保证实时性？

**回答**：

> "我们分数据层和检索层两部分：
>
> **数据层实时性**
> 气象数据通过定时任务从北京局接口拉取：实况每10分钟、逐时每小时、逐天每天3次。数据写入 MySQL，查询时直接读最新记录。
>
> **检索层实时性**
> 知识库更新后需要重新入库（CSV → Embedding → Milvus）。目前是全量重建，194条数据几秒钟完成。后续数据量大了可以做增量更新——只处理新增条目，不重建全量。
>
> **LLM 实时性**
> 每次分析都是实时调用 LLM，不是缓存结果。因为用户反馈内容每次都不同，需要 LLM 针对性分析。"

---

### 10.4 如果检索到的知识和实时数据矛盾怎么办？

**回答**：

> "这是个好问题。举个例子：知识库里有个案例说'温度比预报高5度是因为阳光直射'，但当前天气是阴天。
>
> 我们的处理方式是：
>
> **1. 数据优先级**
> 实时气象数据 > 历史案例。LLM 会同时看到两者，但 Prompt 中明确要求'基于当前数据分析'。
>
> **2. LLM 综合判断**
> LLM 看到实时数据是阴天，就不会引用'阳光直射'的案例，而是找其他原因。
>
> **3. 结果校验**
> 如果 LLM 输出的根因和实时数据明显矛盾（比如说是'阳光直射'但实际是阴天），后处理可以捕获并提示异常。
>
> 这也是为什么我们把实时数据和检索结果都给 LLM——让它自己判断哪些参考有价值。"

### 10.5 你们系统的核心技术难点是什么？怎么解决的？

**回答**：

> "主要有三个难点：
>
> **难点1：中文语义匹配**
> 用户说法多样，'热死了'、'太热了'、'温度偏高'表达的是同一个问题，但关键词完全不同。
>
> 解决：用 BGE 向量模型做语义检索，把文本转成 1024 维向量，计算余弦相似度。语义相近的文本向量距离近，就能匹配上。
>
> **难点2：知识库数据质量**
> 原始 CSV 有脏数据——severity 字段混入了'跨部门'、'数据源'等非严重程度的值；tags 粒度不统一，'气象知识'太模糊。
>
> 解决：数据清洗（severity 只保留 high/medium/low），tags 细化（'气象知识'拆成'体感温度'、'气象原理'、'天气系统'等具体标签）。
>
> **难点3：检索效果评估**
> 怎么知道检索结果好不好？没有标准答案。
>
> 解决：构建评估脚本 + 标注测试集。我们准备了 15 个测试用例，每个用例有期望的模块匹配。跑完计算 Top-K 命中率，分析 Badcase 优化知识库。"

### 10.6 如果让你重新设计，会怎么改进？

**回答**：

> "有几个方向：
>
> **1. 混合检索**
> 现在只用向量检索，可以加 BM25 关键词检索 + 结构化过滤（按 module、tags 过滤），然后用 RRF（Reciprocal Rank Fusion）融合多路结果。工业界标准做法。
>
> **2. Reranker**
> 向量检索召回 Top-20 后，用 Cross-encoder（如 BGE-Reranker）精排，保留 Top-3。比单纯向量检索精度高 10-15%。
>
> **3. Query 改写**
> 用户查询可能太模糊（'热死了'），可以用 LLM 改写成更明确的查询（'温度偏高体感不适'），再检索。
>
> **4. 知识库微调**
> 用业务数据微调 Embedding 模型，让向量更懂我们的领域。比如'闷热'和'高温高湿'在通用模型里可能不够近，微调后能拉近。
>
> **5. 多轮对话**
> 现在是单轮分析，用户追问时没有上下文。可以加对话历史管理，支持多轮交互。"

### 10.7 你们怎么处理检索不到结果的情况？

**回答**：

> "我们的系统有兜底机制：
>
> **1. 双层检索**

> **2. LLM 兜底**

>
> **3. 标记未知**
> 如果 LLM 也无法判断根因，会输出'问题类型: 未知'，建议人工介入。不会强行编造一个答案。
>
> **4. 反馈循环**
> 检索不到的查询会被记录下来，定期分析。如果是高频查询但知识库没有对应案例，就补充进去。"

### 10.11 LLM 调用超时怎么办？你们怎么处理？

**回答**：

> "LLM 超时是我们遇到的主要工程问题之一。分析下来有几种情况：
>
> **超时原因**
> 1. **网络延迟**：API 服务不稳定，偶尔响应慢
> 2. **Prompt 过长**：输入 token 多，模型推理时间长
> 3. **并发限流**：API 有 QPS 限制，高并发时排队
> 4. **模型负载**：高峰期模型服务负载高
>
> **解决方案**
>
> **1. 超时配置 + 重试**
> ```python
> # 配置超时时间
> client = AsyncOpenAI(timeout=30.0)
> # 指数退避重试
> for attempt in range(3):
>     try:
>         response = await client.chat.completions.create(...)
>         break
>     except TimeoutError:
>         await asyncio.sleep(2 ** attempt)
> ```
>
> **2. 异步处理**
> 单条分析同步调用，批量分析用后台任务异步处理。用户提交后立即返回 batch_id，后台慢慢跑。
>
> **3. 降级方案**
> LLM 不可用时，返回基础分析结果（只基于规则判断），不阻塞用户。
>
> **4. 缓存热点**
> 相同反馈内容的分析结果缓存 1 小时，避免重复调用。
>
> **5. 模型选择**
> 简单任务用轻量模型（快、便宜），复杂任务用大模型（慢、贵）。"

### 10.12 你们怎么处理 LLM 返回结果不稳定的问题？

> - 温度值在 -50℃~60℃ 范围内
> - 枚举值在预定义列表中
> - 必填字段不为空
>
> **4. 多次采样 + 投票**
> 重要场景调用 3 次，取众数结果。成本高但稳定性好。
>
> **5. Prompt 优化**
> 给更多示例（few-shot），让模型理解期望的输出格式。"

### 10.19 你们怎么处理 LLM 幻觉问题？

**回答**：

> "LLM 幻觉是指模型编造不存在的事实。我们从预防和检测两方面处理：
>
> **预防措施**
>
> **1. 数据锚定**
> 把真实数据注入 Prompt，让 LLM 基于数据回答：
> ```python
> prompt = f"""
> 基于以下数据回答，不要编造：
> 实况温度：{weather.temp}℃
> 湿度：{weather.humidity}%
> 用户问题：{feedback.content}
> """
> ```
>
> **2. 知识约束**
> 明确告诉 LLM 只用提供的知识：
> ```python
> prompt += """
> 只使用以下知识，不要使用其他知识：
> {knowledge_base}
> """
> ```
>
> **3. 输出校验**
> 检查 LLM 输出是否与输入数据一致：
> ```python
> # 如果 LLM 说"温度 35℃"但实际是 25℃
> if "35℃" in reply and weather.temp != 35:
>     reply = reply.replace("35℃", f"{weather.temp}℃")
> ```
>
> **检测措施**
>
> **1. 事实核查**
> 对比 LLM 输出和原始数据：
> ```python
> def check_hallucination(reply, data):
>     # 提取回复中的数字
>     numbers_in_reply = extract_numbers(reply)
>     # 对比原始数据
>     for num in numbers_in_reply:
>         if num not in data.values():
>             return True  # 可能是幻觉
>     return False
> ```
>
> **2. 多模型交叉验证**
> 用不同 LLM 生成，对比结果：
> ```python
> reply1 = await llm1.generate(prompt)
> reply2 = await llm2.generate(prompt)
> if reply1 != reply2:
>     # 可能有幻觉，人工审核
>     flag_for_review(prompt, reply1, reply2)
> ```
>
> **3. 人工抽检**
> 定期抽查 LLM 输出，标记幻觉案例，优化 Prompt。"

### 10.20 你们用的向量数据库为什么会有锁问题？并发性能怎么解决？

**回答**：

> "我们用的是 Milvus Lite，这是个轻量版本，设计上就不支持高并发。
>
> **为什么会有锁问题？**
>
> Milvus Lite 的架构：
> ```
> 进程A ──┐
>         ├──→ [文件锁] ──→ milvus_weather.db
> 进程B ──┘
> ```
>
> 1. **文件级锁**：Milvus Lite 用文件锁防止多进程同时写入，保证数据一致性
> 2. **单进程设计**：它是嵌入式数据库（类似 SQLite），不是客户端-服务器架构
> 3. **gRPC 限制**：内部用 gRPC 通信，连接数有上限，频繁请求会触发 GOAWAY 错误
>
> **具体错误**
> ```
> DataDirLockedError: another process holds the lock
> GOAWAY with error code ENHANCE_YOUR_CALM: too_many_pings
> ```
>
> **解决方案**
>
> **开发阶段（Milvus Lite）**
> ```python
> # 1. 单例模式，只创建一个客户端
> class MilvusClientSingleton:
>     _instance = None
>
>     @classmethod
>     def get_instance(cls):
>         if cls._instance is None:
>             cls._instance = MilvusClient("milvus.db")
>         return cls._instance
>
> # 2. 查询前自动 load
> client.load_collection("weather_feedback")
>
> # 3. 连接池管理
> # 避免频繁创建/销毁连接
> ```
>
> **生产阶段（Milvus Standalone/Distributed）**
> ```python
> # 切换到服务端架构
> client = MilvusClient(uri="http://milvus-server:19530")
>
> # 支持：
> # - 多进程并发访问
> # - 连接池管理
> # - 水平扩展
> # - 高可用
> ```
>
> **版本对比**
>
> | 维度 | Milvus Lite | Milvus Standalone | Milvus Distributed |
> |------|-------------|-------------------|-------------------|
> | 部署 | 嵌入式 | Docker 单机 | K8s 集群 |
> | 并发 | 单进程 | 百级并发 | 万级并发 |
> | 数据量 | <100 万 | <1000 万 | 亿级 |
> | 适用 | 开发测试 | 小规模生产 | 大规模生产 |
> | 锁问题 | 有 | 无 | 无 |

### 11.6 面试话术

**问：你们系统的日均反馈量是多少？怎么应对高并发？**

> "日均反馈量 500-3000 条，重大天气时 1-3 万条。
>
> 系统设计上，我们用了几个方案应对：
>
> **1. 消息队列削峰**
> API 接入后发到 Kafka，同步返回分析 ID，消费者异步处理。突发流量时队列缓冲，不会压垮系统。
>
> **2. 消费者集群**
> 多实例并行消费，K8s HPA 根据队列积压自动扩缩容。积压超过 100 条自动加实例。
>
> **3. 优先级队列**
> 预警相关反馈发到高优 Topic，优先处理。体验问题发到低优 Topic，慢慢处理。
>
> **4. 批量处理**
> 合并多条反馈一次调 LLM，降低调用次数和成本。
>
> **5. 缓存复用**
> 相同城市 + 相同天气条件的结果缓存，避免重复计算。
>
> 这样单机可以处理 1 万条/天，集群可以处理 10 万条/天。"

**问：怎么保证高优先级反馈优先处理？**

> "我们用优先级队列：
>
> 1. **识别优先级**：根据反馈内容判断，包含'预警'的标记为高优
> 2. **分 Topic 存储**：高优反馈发到 `feedback_high`，普通反馈发到 `feedback_normal`
> 3. **优先消费**：消费者优先从高优 Topic 拉取数据
> 4. **插队机制**：高优反馈可以插队到队列前面
>
> 这样预警相关的反馈可以秒级处理，普通反馈分钟级处理。"

**问：怎么降低成本？**

> "几个方案：
>
> 1. **批量处理**：合并多条反馈一次调 LLM，降低调用次数
> 2. **缓存复用**：相同问题直接返回缓存结果
> 3. **模型分级**：简单问题用轻量模型，复杂问题用大模型
> 4. **Prompt 精简**：优化 Prompt 长度，减少 Token 消耗
>
> 优化后单次成本从 0.05 元降到 0.01 元，日处理 3000 条只需 30 元。"

#### 技术实现

```python
class FinancialPlanningAssistant:
    def __init__(self):
        self.milvus = MilvusClient()      # 规划案例向量库
        self.es = Elasticsearch()          # 税法条款、产品信息
        self.neo4j = Neo4jClient()        # 税务规则、投资关系图谱
        self.llm = LLMService()           # 豆包大模型
        self.calc = TaxCalculator()       # 税务计算引擎

    async def plan_retirement(self, client_profile):
        age = client_profile["age"]
        assets = client_profile["assets"]
        target_age = client_profile["retirement_age"]

        # 1. 图谱查询相关规则
        rules = await self.neo4j.query(f"""
            MATCH (rule:Rule)
            WHERE rule.min_age <= {age} AND rule.max_age >= {age}
            RETURN rule
        """)

        # 2. ES 检索类似案例
        cases = await self.es.search({
            "query": {
                "bool": {
                    "must": [{"match": {"type": "retirement_plan"}}],
                    "filter": [
                        {"range": {"client_age": {"gte": age-5, "lte": age+5}}},
                        {"range": {"assets": {"gte": assets*0.8, "lte": assets*1.2}}},
                    ]
                }
            }
        })

        # 3. 计算税务影响
        tax_impact = self.calc.calculate_retirement_tax(
            age=age, assets=assets, retirement_age=target_age
        )

        # 4. LLM 生成建议
        return await self.llm.generate(
            f"客户{age}岁，资产{assets}万，想{target_age}岁退休",
            context={"rules": rules, "cases": cases, "tax_impact": tax_impact}
        )
```

**问：知识图谱怎么构建？**

> "两个路径：
>
> **1. 结构化数据导入**
> 医药：药物-疾病-不良反应关系 → 从医学知识库导入
> 金融：税务规则-投资产品关系 → 从法规文档导入
>
> **2. 非结构化数据抽取**
> 文档 → LLM/NER 实体识别 → 关系抽取 → 图谱入库
>
> 维护上，增量更新为主——新文档进来实时抽取，定期全量校验。"

**问：三路检索怎么融合？**

> "用 RRF（Reciprocal Rank Fusion）算法：
>
> ```python
> def rrf_fusion(results_list, k=60):
>     scores = {}
>     for results in results_list:
>         for rank, doc in enumerate(results):
>             scores[doc.id] = scores.get(doc.id, 0) + 1 / (k + rank)
>     return sorted(scores.items(), key=lambda x: -x[1])
> ```
>
> 或者加权融合：
> - 向量检索权重 0.4（语义相关性）
> - ES 检索权重 0.3（精确匹配）
> - 图谱检索权重 0.3（关系推理）
>
> 权重可以根据业务场景调整。"

## 十三、技术选型：ES vs 知识图谱 vs 向量数据库

### 13.1 一句话总结

| 技术 | 解决什么问题 | 一句话 |
|------|-------------|--------|
| **Elasticsearch** | 海量数据的结构化查询和全文检索 | "从百万条数据里按条件筛选" |
| **知识图谱** | 实体关系的存储和推理 | "A和B有什么关系，能推导出什么" |
| **向量数据库** | 语义相似度匹配 | "找意思相近的内容" |

### 13.2 核心能力对比

| 能力 | ES | 知识图谱 | 向量数据库 |
|------|-----|---------|-----------|
| 全文检索 | ✅ 强 | ❌ | ❌ |
| 结构化过滤 | ✅ 强 | ✅ 中 | ❌ |
| 语义匹配 | ❌ | ❌ | ✅ 强 |
| 关系推理 | ❌ | ✅ 强 | ❌ |
| 聚合统计 | ✅ 强 | ❌ | ❌ |
| 模糊匹配 | ✅ 中 | ❌ | ✅ 强 |
| 多跳查询 | ❌ | ✅ 强 | ❌ |

### 13.3 数据结构对比

| 技术 | 数据结构 | 示例 |
|------|---------|------|
| **ES** | 文档（JSON） | `{"content": "温度偏差", "city": "北京", "date": "2024-01-15"}` |
| **知识图谱** | 图（节点+边） | `(北京)-[:发布了]->(高温预警)-[:影响]->(体感温度)` |

## 十四、Function Calling 设计与实现

### 14.1 什么是 Function Calling

Function Calling 是 LLM 的一种能力：LLM 不直接回答问题，而是决定调用哪个外部函数、传什么参数，拿到结果后再生成回答。

```
用户输入 → LLM 理解意图 → 选择 Function → 生成参数 → 执行 → 返回结果 → LLM 生成回答
```

### 14.2 我们为什么用 Function Calling

| 场景 | 用 Skill | 用 Function Calling |
|------|----------|---------------------|
| 天气反馈 | ✅ 流程固定，1次 LLM | ❌ 过度设计 |
| 指数反馈 | ❌ 指数类型多，流程不固定 | ✅ LLM 动态选择 |

**天气反馈**：每次都是查实况→查逐时→查逐天→查预警→算体感→搜知识，流程固定，用 Skill 更高效。

**指数反馈**：有 7 种指数（穿衣/紫外线/中暑/感冒/运动/舒适度/出行），用户可能只反馈其中一种，需要 LLM 判断是哪个指数再查对应数据。

### 14.3 Function Calling 设计

#### 4 个 Function

| Function | 说明 | 数据源 |
|----------|------|--------|
| `get_index_data` | 获取单个指数数据 | live_index 表 |
| `get_index_rules` | 获取指数计算规则 | IndexEngine 规则库 |
| `get_weather_for_index` | 获取计算用天气数据 | weather_ff 表 |
| `get_all_indices` | 获取所有指数 | live_index 表 |

#### 调用流程

```
用户反馈: "运动指数说适宜，但我差点中暑了"
    ↓
LLM 第1次调用：识别指数类型 + 选择 Function
    → 调用 get_index_data("海淀", "2026-07-17", "运动")
    → 调用 get_index_rules("运动")
    → 调用 get_weather_for_index("海淀", "2026-07-17")
    ↓
执行 Function，返回结果
    ↓
LLM 第2次调用：分析原因 + 生成回复
    → "运动指数计算未纳入湿度因素，当日湿度较低，
       体感温度偏高，易引发中暑..."
```

### 14.4 关键词路由

```python
INDEX_KEYWORDS = ["指数", "穿衣", "运动", "紫外线", "中暑", "感冒",
                  "舒适度", "出行", "适宜", "不适宜", "闷热", "炎热", "寒冷"]

def _is_index_feedback(content: str) -> bool:
    return any(kw in content for kw in INDEX_KEYWORDS)

# 路由逻辑
if _is_index_feedback(feedback.content):
    → Function Calling 流程（2次 LLM）
else:
    → Skill 流程（1次 LLM）
```

**为什么用关键词不用 LLM**：避免额外一次 LLM 调用，关键词判断够用。

### 14.5 API 接口设计

| 接口 | 说明 |
|------|------|
| `/api/v1/agent/analyze` | 自动路由（关键词判断走 Skill 或 FC） |
| `/api/v1/agent/analyze-weather` | 强制走 Skill（天气反馈） |
| `/api/v1/agent/analyze-index` | 强制走 Function Calling（指数反馈） |

### 14.6 面试话术

**问：什么是 Function Calling？你们怎么用的？**

> "Function Calling 是 LLM 的一种能力：LLM 不直接回答，而是决定调用哪个外部函数、传什么参数，拿到结果后再生成回答。
>
> 我们在生活指数场景用了 Function Calling。有 7 种指数（穿衣/紫外线/中暑等），用户可能只反馈其中一种，需要 LLM 判断是哪个指数再查对应数据。
>
> 比如用户说'运动指数说适宜但我差点中暑'，LLM 识别是运动指数问题，调用 Function 查询指数数据、计算规则、天气数据，然后分析：运动指数计算未纳入湿度因素，当日湿度较低导致体感温度偏高。"

> "核心区别是谁决定调什么。

**问：为什么不全部用 Function Calling？**

> "成本和延迟。
>
> Skill 架构：1次 LLM 调用，3-5秒，0.01元
> Function Calling：2次 LLM 调用，6-10秒，0.02元
>
> 天气反馈流程固定，每次都是查实况→查逐时→查逐天→查预警，用 Skill 更快更省。
>
> 指数反馈流程不固定，需要 LLM 判断是哪个指数，才用 Function Calling。
>
> 技术选型要贴合业务实际，不要过度设计。"

**问：怎么判断用 Skill 还是 Function Calling？**

> "看两个因素：
>
> **1. 流程是否固定**
> 固定 → Skill
> 不固定 → Function Calling
>
> **2. 参数是否明确**
> 从输入直接提取 → Skill
> 需要从自然语言理解 → Function Calling
>
> 我们天气反馈流程固定、参数明确（location/time 直接提取），用 Skill。
> 指数反馈流程不固定（7种指数）、需要 LLM 判断是哪种，用 Function Calling。"

**问：关键词路由会不会误判？**

> "会，但概率低。我们用的关键词是指数相关的专业术语（穿衣、运动、紫外线、中暑等），普通天气反馈很少包含这些词。
>
> 如果误判了也没关系：
> - 指数反馈误判为天气 → LLM 仍然能分析，只是不用 Function
> - 天气反馈误判为指数 → LLM 调用 Function 查不到数据，会降级处理
>
> 生产环境可以加日志监控误判率，持续优化关键词列表。"
