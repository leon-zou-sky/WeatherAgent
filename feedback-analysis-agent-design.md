## 一、项目背景

### 1.1 业务背景

气象数据服务面向亿级用户提供天气预报、预警、空气质量等数据产品。每周收到大量用户负反馈，包括"天气不准"、"没有预警"、"空气质量不准"等问题。

### 1.2 现状痛点

| 痛点 | 描述 | 影响 |
|------|------|------|
| 人工排查耗时 | 每个反馈需要 1-2 小时排查 | 人力成本高 |
| 响应慢 | 用户等待 2-3 天才有回复 | 用户体验差 |
| 链路复杂 | 数据源→采集→处理→存储→发布→用户 | 定位困难 |
| 产品多样 | 温度、风速、降水、预警、AQI 等 | 排查工作量大 |
| 知识流失 | 经验在人脑中，人员离职就丢失 | 培训成本高 |

### 1.3 项目目标

构建智能 Agent 系统，实现负反馈自动分析和链路排查，提升排查效率和用户体验。

**核心价值**：
- 排查时间：2 小时 → 5 分钟
- 人力成本：降低 90%
- 响应速度：提升 100 倍

## 四、技术方案

### 4.1 技术选型

| 组件 | 技术选型 | 说明 |
|------|----------|------|
| Agent 框架 | 自研编排 | 直接编排，不需要 LangChain |
| LLM | 豆包大模型（Ark） | OpenAI 兼容接口 |
| RAG | Milvus Lite + BGE-large-zh | 向量检索 + 关键词兜底 |
| 后端框架 | FastAPI | 异步支持、性能好 |
| 数据库 | MySQL + Redis | 元数据 + 缓存 |
| Embedding | BGE-large-zh-v1.5 | 中文优化、1024 维 |

### 4.2 为什么不用 LangChain

| 维度 | LangChain | 自研编排 |
|------|-----------|---------|
| 灵活性 | 框架约束 | 完全可控 |
| 性能 | 有额外开销 | 最优 |
| 学习成本 | 高 | 低 |
| 调试 | 困难 | 容易 |
| 适用场景 | 复杂链路 | 简单编排 |

**结论**：我们的场景是固定编排（收集数据 → LLM 分析），不需要 LangChain 的动态路由能力。

### 4.3 为什么选 BGE-large-zh

| 维度 | BGE-large-zh | 其他选项 |
|------|--------------|---------|
| 中文优化 | ✅ 智源研究院专为中文训练 | 多语言模型中文效果一般 |
| 开源免费 | ✅ MIT 协议 | OpenAI Embedding 要付费 |
| 本地运行 | ✅ 不依赖外部 API | API 类有网络延迟 |
| 维度 | 1024 维，精度高 | 小模型 384/768 维精度低 |
| 社区活跃 | ✅ HuggingFace 下载量高 | 冷门模型文档少 |

### 4.5 Function Calling 设计（生活指数分析）

#### 架构说明

生活指数分析采用 Function Calling 架构，与普通天气反馈的 Skill 架构形成互补：

```
用户反馈
    ↓
关键词路由
    ↓
┌─────────────────┬─────────────────┐
│ 天气反馈        │ 指数反馈        │
│ (Skill 架构)    │ (FC 架构)       │
└─────────────────┴─────────────────┘
    ↓                    ↓
固定调用所有 Skill    LLM 动态选择 Function
    ↓                    ↓
1次 LLM 调用          2次 LLM 调用
```

| 出行 | 出行适宜度 | 天气、温度、AQI、风力 |

#### 调用流程

```
用户反馈: "运动指数说适宜，但我差点中暑了"
    ↓
LLM 第1次调用：识别指数类型 + 选择 Function
    → 调用 get_index_data("海淀", "2026-07-17", "运动")
    → 调用 get_index_rules("运动")
    → 调用 get_weather_for_index("海淀", "2026-07-17")
    ↓
执行 Function，返回结果
    ↓
LLM 第2次调用：分析原因 + 生成回复
    → "运动指数计算未纳入湿度因素，当日湿度较低，
       体感温度偏高，易引发中暑..."
```

#### 关键词路由


### 4.6 Prompt 设计

#### 系统提示词

```
你是气象数据质量分析专家，负责分析用户反馈的气象数据问题。

## 你的能力：
1. 分析用户反馈，判断问题类型
2. 检查整个数据链路，定位问题环节
3. 计算体感温度，解释用户感受
4. 检索相似案例，参考历史方案
5. 生成分析报告和回复内容

## 重要规则 - 信息分层：

### 内部信息（用于分析，不告诉用户）：
- 链路检查结果（数据延迟、系统状态）
- 数据源状态（气象站、数据库）
- 内部技术细节

### 用户回复（只包含）：
- 实际气象数据
- 体感温度分析
- 气象原理解释
- 可操作建议

## reply_content 生成规则：
1. 如果有生效预警，必须在回复中说明当前预警状态
2. 如果用户反馈与预警相关，结合预警信息解释
3. 不要只给数据，要解释原因
4. 语气友好，像真人客服
```

## 六、RAG 知识库设计

### 6.1 核心理念

**RAG 的效果 = 数据质量 × 40% + 标签/分类 × 25% + Embedding 模型 × 20% + 检索策略 × 15%**

RAG 的核心不是算法，是**数据工程**。算法（BGE、Milvus）只是工具，数据质量才是决定性因素。

### 6.2 知识库数据结构

#### 字段设计（7 字段）

| 字段 | 说明 | 用途 | 重要度 |
|------|------|------|--------|
| `module` | 产品模块（14类） | Layer 1 结构化过滤 | ⭐⭐⭐⭐⭐ |
| `problem_pattern` | 问题模式（6类） | Layer 2 分类匹配 | ⭐⭐⭐⭐ |
| `problem_desc` | 问题描述 | 向量检索主字段 | ⭐⭐⭐⭐⭐ |
| `root_cause` | 根因分析 | 向量检索 + LLM 参考 | ⭐⭐⭐⭐ |
| `solution` | 解决方案 | 返回给 LLM | ⭐⭐⭐⭐ |
| `tags` | 语义标签 | 向量检索增强 | ⭐⭐⭐ |
| `severity` | 严重程度 | 排序加权 | ⭐⭐ |

#### Embedding 拼接策略

```python
# 多字段组合向量化，提升语义丰富度
embed_text = problem_desc + root_cause + solution + tags
```

**为什么组合？**
- 单字段："温度不准" → 语义模糊，可能匹配到各种温度相关
- 组合后："温度不准 + 城市热岛效应 + 引入高分辨率模型 + 温度" → 语义精确，匹配到具体问题和方案

### 6.3 三层索引架构

```
┌──────────────────────────────────────────────────────────┐
│  Layer 1: 产品模块（module）- 14 类                       │
│  实况/逐天预报/逐小时预报/短临降水/天气预警/              │
│  空气质量/生活指数/格点可视化/台风预报/定位/              │
│  App体验/数据时效/潮汐海浪/航空专业气象                   │
└──────────────────────────────────────────────────────────┘
                          ↓
┌──────────────────────────────────────────────────────────┐
│  Layer 2: 问题模式（problem_pattern）- 6 类               │
│  数据偏差/时空误差/时效延迟/认知偏差/体验缺陷/产品设计    │
└──────────────────────────────────────────────────────────┘
                          ↓
┌──────────────────────────────────────────────────────────┐
│  Layer 3: 语义标签（tags）                                │
│  温度/降雨/风力/体感温度/气象原理/预警理解/...            │
└──────────────────────────────────────────────────────────┘
```

**分层的作用**：
- **检索前过滤**：先按 module 粗筛，缩小候选集
- **检索后排序**：severity=high 的优先展示
- **聚类分析**：发现高频问题类型和知识盲区

### 6.4 数据构建方法

#### 方法 1：模板变体生成

```python
# 基础模板
("温度比预报高{N}度", "城市热岛效应未充分建模", "引入高分辨率城市热岛模型")

# 生成变体
for n in [2, 3, 5, 7, 10]:
    → "温度比预报高2度"
    → "温度比预报高3度"
    → "温度比预报高5度"
    ...
```

**作用**：覆盖用户不同的表述方式，提升语义匹配概率。

#### 方法 2：模块均衡扩充

```
扩充前分布（不均衡）：
  实况: 80 条 (41%)  ← 太多
  逐天预报: 5 条 (3%) ← 太少
  逐小时预报: 3 条 (2%) ← 太少

扩充后分布（均衡）：
  实况: 210 条 (25%)
  逐天预报: 85 条 (10%)
  逐小时预报: 49 条 (6%)
  格点可视化: 79 条 (9%)
  空气质量: 78 条 (9%)
```

**方法**：识别知识盲区 → 定向补充 → 重新评估

#### 方法 3：产品设计问题挖掘

```python
# 产品设计问题模板
("城市级数据精度不够", "使用城市级数据而非格点数据", "接入经纬度格点数据提升精度")
("没有体感温度数据", "产品未展示体感温度", "增加体感温度展示")
("没有空气质量地图", "产品功能缺失", "增加空气质量分布地图")
```

**来源**：
- 用户投诉分析：哪些问题是因为产品设计导致的？
- 竞品对比：竞品有但我们没有的功能
- 技术评审：数据精度、覆盖范围、更新频率等

#### 方法 4：Badcase 闭环优化

```
Badcase 分析流程：
  1. 评估测试 → 发现 Badcase
  2. 分析原因 → 为什么匹配错？
  3. 补充条目 → 增加更精确的变体
  4. 重新评估 → 验证修复效果

案例：
  问题："未来两小时预报不准" 匹配到 "逐天预报"
  原因：知识库缺少 "两小时内" 的逐小时预报案例
  解决：补充 "两小时内天气预报偏差"、"一两小时预报不准" 等条目
  效果：Top-3 命中率从 76% → 100%
```

#### 方法 5：语义聚类

```
聚类维度：
├── 按模块聚类：14 个产品模块
├── 按问题模式聚类：6 种问题类型
├── 按标签聚类：温度/风速/降水/...
└── 按严重程度聚类：high/medium/low

聚类的作用：
├── 发现知识盲区：哪些模块案例太少？
├── 去重：相似案例合并
├── 分层检索：先粗类再细类
└── 质量评估：同类案例一致性检查
```

### 6.5 检索策略

#### 双层检索：向量优先 + 关键词兜底

```python
async def search_knowledge(query: str, top_k: int = 3):
    # 第一层：向量检索（语义匹配）
    results = _vector_search(query, top_k)
    
    # 第二层：关键词兜底（精确匹配）
    if not results:
        results = _keyword_search(query, top_k)
    
    return results
```

**为什么双层？**
- 向量检索：擅长语义匹配（"热死了" → "体感温度"案例）
- 关键词检索：擅长精确匹配（"PM2.5" → "PM2.5"案例）
- 互补：向量检索失败时，关键词检索兜底

#### 置信度分层

```python
if results[0].score > 0.85:
    # 高置信度：直接返回 Top-1
    return [results[0]]
else:
    # 低置信度：返回 Top-3 给 LLM 更多参考
    return results
```

## 十三、MCP 服务设计

### 13.1 MCP 概述

MCP（Model Context Protocol）是 Anthropic 推出的开放协议，用于 LLM 与外部工具/数据源的标准连接。本项目将 Skills 层能力通过 MCP 协议暴露，使客服/运营人员可通过 Trae、Cursor、Claude Desktop 等 AI 客户端直接调用气象分析能力。

**核心价值**：零前端开发，LLM 就是界面，打字就能用。

### 13.2 MCP 架构

```
┌──────────────────────────────────────────────────────┐
│                 MCP Clients                           │
│   Trae / Cursor / Claude Desktop / 自定义 Agent       │
└────────────────────┬─────────────────────────────────┘
                     │  MCP 协议 (stdio / SSE)
                     ▼
┌──────────────────────────────────────────────────────┐
│           WeatherAgent MCP Server                     │
│                                                       │
│   @mcp.tool()           @mcp.resource()               │
│   ┌─────────────────┐   ┌───────────────────┐        │
│   │ 6 个 Tools       │   │ 2 个 Resources    │        │
│   └────────┬────────┘   └────────┬──────────┘        │
│            │                     │                    │
│            ▼                     ▼                    │
│   复用 app/skills/ 下的已有函数（零改动）              │
└──────────────────────────────────────────────────────┘
                     │
                     ▼
┌──────────────────────────────────────────────────────┐
│   MySQL + Milvus + LLM API（已有数据层）              │
└──────────────────────────────────────────────────────┘
```

### 13.3 MCP Tools 定义

| Tool 名称 | 角色 | 说明 | 对应 Skill |
|-----------|------|------|-----------|
| `query_weather` | 客服 | 查询城市实况气象数据 | `weather.query_weather_data` |
| `query_alert` | 客服 | 查询当前生效的气象预警 | `alert.query_alert_data` |
| `search_knowledge` | 客服 | 检索知识库相似案例 | `knowledge.search_knowledge` |
| `analyze_feedback` | 客服 | 分析负反馈生成报告 | `agent.core.smart_analyze` |
| `check_pipeline` | 运营 | 检查数据链路健康状态 | `pipeline.check_pipeline` |
| `get_monitor_overview` | 运营 | 获取反馈监控概览 | `alert_monitor._get_stats` |

### 13.4 MCP 传输模式

| 模式 | 适用场景 | 启动方式 |
|------|---------|---------|
| stdio | 本地开发、Claude Code | `python mcp_main.py` |
| SSE | 生产部署、多客户端 | `mcp.run(transport='sse', host='0.0.0.0', port=9000)` |

### 13.6 客户端配置

**Trae / Cursor（SSE 模式）**：

```json
{
  "mcpServers": {
    "weather-agent": {
      "url": "http://服务器IP:9000/sse"
    }
  }
}
```

**Claude Code（stdio 模式）**：

```json
{
  "mcpServers": {
    "weather-agent": {
      "command": "python",
      "args": ["mcp_main.py"]
    }
  }
}
```

### 13.7 使用场景

**客服场景**：
```
客服输入："用户反馈北京温度不准，帮我查一下"
Claude/Trae 自动调用：
  → query_weather("北京")
  → search_knowledge("温度不准")
  → 返回实况数据 + 相似案例 + 建议回复
```

**运营场景**：
```
运营输入："看下最近24小时反馈情况"
Claude/Trae 自动调用：
  → get_monitor_overview(24)
  → 返回反馈量、类型分布、异常告警
```

## 十四、风险评估

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| LLM 幻觉 | 生成不准确内容 | 数据锚定、输出校验、人工审核 |
| LLM 超时 | 响应慢 | 超时重试、异步处理 |
| Token 成本 | 费用高 | Prompt 精简、缓存复用、模型分级 |
| 向量数据库并发 | 性能瓶颈 | 生产用 Milvus Standalone |
| 知识库质量 | 检索效果差 | 持续补充、Badcase 闭环 |

## 十五、总结

本项目通过构建智能 Agent 系统，实现负反馈自动分析和链路排查，解决人工排查耗时、响应慢、链路复杂等痛点。

**核心架构**：Skill + Agent 混合架构
- Agent 层：负责智能决策（理解反馈、综合分析、生成回复）
- Skill 层：负责固定功能（数据查询、知识检索、体感计算）

**技术亮点**：
- RAG 知识库：Milvus + BGE-large-zh，Top-1 命中率 93.3%
- 信息分层：内部诊断信息不暴露给用户
- 生产级功能：批量分析、审核流程、问题预警

**核心价值**：
- 排查时间从 2 小时降到 5 分钟
- 人力成本降低 90%
- 响应速度提升 100 倍
